The path forward

  • Innovation and technology advances are enabling move towards more appropriate form factors
  • We will be able to use the interface device that is most suitable for a given task.
  • Several axes at play:
    • Social vs asocial tasks
    • Consuming versus producing data
    • Environmental constraints on form factor
  • Software services being decoupled from specific devices.

Furious innovation

  • There will be a shift towards more cross-platform solution - content producers will demand it.
  • Build time: PhoneGap, Tabris, Quark App Studio, GWT Mobile
  • Languages: Dart, TypeScript, CoffeeScript
  • Runtimes: Chrome/Firefox OS, SPDY
  • Frameworks: Sencha, jQuery, Tiggzi, Widgetbox, appMobi
  • Nobody knows the answer yet.

Eclipse impact: servers

  • Server side will continue to be vital
  • Java and OSGi/Equinox still very compelling
  • Other alternatives: PHP, Ruby/Rails, Go, ASP, Perl, Node
  • Eclipse innovation: Equinox, Jetty, Virgo, Gemini, Gyrex
  • Don't marry your client to a particular server technology

Eclipse impact: tools

  • Programming is inherently social and the data will want to be in the cloud
  • Desktop still well suited for many tasks
  • Gradual evolution toward web (bugzilla, gerrit, stackoverflow, hudson)
  • Eclipse Platform 4 will enable the transition
  • Eclipse innovation: Orion, WikiText, Hudson, M2M
  • Make your tooling functionality loosely coupled

Eclipse impact: rich clients

  • Highly depends on your functionality and end users
  • Eclipse 4 RCP opens doors: CSS styling, pluggable renderer
  • Java clients will become niche
  • Paths forward for existing code: move to server, cross compile to client
  • Eclipse innovation: Eclipse Platform 4, Scout, RAP mobile, Riena
  • Make your functionality loosely coupled

Different value propositions

  • Foundations:
    • Strong legal controls
    • Predictable schedules
    • Heavy infrastructure
    • Vendor neutral governance
  • Community forges:
    • No legal process
    • Freedom from fixed schedules
    • Just enough infrastructure
    • No governance

Process diet

  • Streamline legal and other processes
  • Reduce time to create a project
  • Click-through committer agreements
  • Embrace the fork: it is ok if we don't all agree
  • Simplify contribution through Git, Gerrit, CBI
  • Enable PMCs and working groups to add process as needed

Barriers to contribution

  • Fighting perception platform is "good enough"
  • If we don't contribute, someone else will
  • No reasonable way to contribute on a small scale
    • It takes a long time to build up deep expertise
    • Can't respond to changing business needs
    • Too expensive to have expertise everywhere
  • No company big enough to sell sponsored development
  • Market failure: public goods

How to increase contribution

  • Guilt
  • Charity
  • State control
  • Forks and process diet

Creating a market for change

  • A huge demand for changes, enhancements, bug fixes
  • Can't sell the software, but can sell the service of changing the software
  • Long Term Support for feature development
  • Use forks to handle conflicting requirements (Git, CBI)

Fundamental value

Strip away all the current technology and there is fundamental value in Eclipse that can endure any technology cycle: Eclipse as *the* place for commercial friendly open source technology. IP clean, transparent development process, predictable schedule, commercial friendly license, vendor neutral governance.

Conclusions

  • Massive changes in how people consume/use software
  • Evolution away from monolithic platforms and apps
  • Smaller focused services, heterogenous devices
  • Community needs to scale processes to suit different projects
  • Eclipse is the best place to do commercial friendly open source
